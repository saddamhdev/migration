## What is **Profile** (in business/domain terms)

> **Profile = a user‚Äôs personal and descriptive information**
> (who the user *is*, not how they log in or pay)

It answers questions like:

* Who is this user?
* How should they be displayed?
* How can we contact them?

---

## Typical responsibilities of **Profile**

### ‚úÖ What belongs in Profile

**Personal data**

* full name
* display name
* date of birth
* gender (if required)

**Contact info**

* email (sometimes)
* phone number
* address

**Public / UI data**

* profile photo
* bio / description
* language preference
* timezone

**User preferences**

* notification preferences
* UI theme
* regional settings

---

## ‚ùå What does NOT belong in Profile

This is where many systems go wrong.

| Feature            | Why NOT profile      |
| ------------------ | -------------------- |
| Login / password   | Security domain      |
| JWT / sessions     | Identity/Auth domain |
| Billing info       | Financial domain     |
| Subscription plans | Commercial domain    |
| Roles/permissions  | Authorization domain |

Profile should be **boring data**, not critical logic.
